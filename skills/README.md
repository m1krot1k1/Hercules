# Skills

Workflow skills описывают повторяемые сценарии и паттерны, которыми пользуются агенты. Каждый skill — файл `SKILL.md` в своём каталоге.

## Реестр skills

| Skill | Назначение | Когда использовать |
|-------|-----------|-------------------|
| **agent-prompt-quality** | Качество промптов при делегировании | Составление `Task(...)` вызовов: AC, Voice/Stop when, язык |
| **agent-quality-pipeline** | Эффективный пайплайн | Задачи с дорогими шагами: dedup, наблюдаемость, таймауты |
| **agent-system-navigation** | Навигация по 33 агентам | Decision tree для выбора агента, паттерны оркестрации |
| **autonomous-execution** | DUA-детекция и act-first стойка | Автономное исполнение без бюрократических блоков |
| **multi-pass-autonomy** | Многоходовая «автономность» | Параллельные Task как голоса, раунды сторонник/критик/арбитр |
| **orchestrator** | Быстрый вызов оркестрации | Триггеры и контракт рекурсивной оркестрации |
| **project-plan-dot-plan** | Каталог `.plan/` | Todos, журнал, файлы по направлениям, таймстемпы |
| **repo-task-proof-loop** | Доказуемый цикл | Крупные фичи: spec freeze → build → evidence → verdict |
| **specialist-discovery** | Маппинг задач → агентов | Таблица всех 33 агентов с «когда использовать» |
| **start-workflow** | Паттерны /start | Когда использовать start vs прямой вызов специалиста |
| **subagent-factory** | Создание агентов | Новый пакет: agent .md + rules .mdc + optional skill |
| **thinking-checkpoints** | STOP-and-think gates | 5 обязательных чекпоинтов: перед декомпозицией, делегированием, каждые 3-5 вызовов, перед синтезом, перед ответом |

## New (Architecture Optimization)

| Skill | Назначение | Когда использовать |
|-------|-----------|-------------------|
| **circuit-breaker** | Предотвращение каскадных сбоев | При вызове ненадёжных внешних сервисов или MCP-инструментов |
| **dynamic-discovery** | Динамическая маршрутизация к специалистам | Когда набор доступных агентов меняется во время выполнения |
| **context-propagation** | Структурированный контекст-конверт | При передаче контекста между агентами с версионированием |
| **dag-validation** | Валидация графа зависимостей | Перед запуском задач с зависимостями для проверки циклов |
| **caid-worktrees** | Изоляция через Git worktree | При параллельной работе над независимыми задачами |
| **telemetry-pipeline** | Сбор и анализ метрик | Для мониторинга производительности и отладки агентных пайплайнов |
| **model-routing** | Мульти-модельная оптимизация затрат | Когда нужно выбрать модель по стоимости/качеству для задачи |

## Связь со структурой

- Канонические пути этого репозитория: `agents/`, `rules/`, `docs/`, `skills/`.
- В целевом проекте эти каталоги обычно разворачиваются под `.cursor/`.
- Мастер-правило со ссылками на skills: `rules/specialists.mdc`.

## Когда создавать новый skill vs расширять существующий

**Создать новый skill** (`skills/<name>/SKILL.md`), если:
- Это самостоятельный повторяемый workflow со своим жизненным циклом (например, `repo-task-proof-loop`).
- Навык охватывает несколько шагов и используется в разных контекстах.
- Логика отличается настолько, что смешение с существующим skill затруднит понимание.

**Расширить существующий skill**, если:
- Это вариант или подпаттерн уже существующего workflow.
- Шаги идентичны, меняется только контекст применения.
- Правки небольшие и не нарушают логику существующих пользователей skill.

**Шаблон нового SKILL.md:**
```yaml
---
name: <kebab-case-name>
description: <одна строка: что делает skill и когда применять>
---
```
Затем разделы: `## Purpose`, `## Когда использовать`, `## Шаги`, `## Policy Reference` (если реализует политику из rules/).

Полная инструкция: [docs/MAINTENANCE_RUNBOOK.md](../docs/MAINTENANCE_RUNBOOK.md#создание-нового-skill)
